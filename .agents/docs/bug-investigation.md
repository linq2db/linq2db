# Bug investigation & fixing

Situational rules for reproducing, scoping, and fixing reported bugs / regressions. The one-line triggers live in [`agent-rules.md`](agent-rules.md) → *Investigating & fixing bugs*; this doc holds the detail and the war-stories that justify each rule. Read it when one of those triggers fires.

## "Fix or disable" cleanup issues — attempt the fix first

When an issue offers a choice — *fix the sites, or disable / suppress the rule* (analyzer-rule cleanups, lint-debt batches like #5532) — do **not** open with a disable-everything plan. Work rule-by-rule: attempt the genuine fix, and reach disable/suppress only after *demonstrating* the fix is infeasible — breaking public/interface API, the needed API absent on a target TFM, the only available fix a behavioral no-op / analyzer appeasement, or every current site a false positive. Put that evidence in the per-rule proposal. Disable is the documented fallback, not the lead. (Corrected mid-#5532, after an opening plan proposed disabling 9 of 10 rules.)

Analyzer-rule mechanics — `.editorconfig` layout, full-solution verification, site enumeration, bulk code-fix application — live in [`analyzer-rules.md`](analyzer-rules.md).

## "Regression after switching package X→Y" reports

When a bug report blames a package swap ("worked with package X, broke after switching to Y"), verify the named package actually contains the relevant code before treating the swap as the cause. Many linq2db packages are thin satellites — e.g. `linq2db.Extensions` and the older `linq2db.AspNet` are DI / logging only, with zero query-translation code. A translation / SQL-generation "regression" that coincides with such a swap is almost always a **core `linq2db` version change** that rode along with it (often a major-version upgrade), not the package itself. Pin the actual core version on both sides before reproducing — and reproduce against that core version, not just current master (#5560 was reported on a swap to `linq2db.Extensions` but the real variable was the v5→v6 core upgrade; it didn't reproduce on 6.3.0 or master at all).

## Reproducing a reported regression — confirm HEAD contains the change first

Before building a repro for a bug attributed to a specific merged PR/commit, confirm the working tree actually contains that change. A long-lived branch (e.g. `infra/*`) can sit weeks behind `origin/master`, so a regression introduced by a recent PR won't reproduce locally and you'll burn time chasing a ghost — building elaborate offline repros against code that predates the bug. Check with `git merge-base --is-ancestor <pr-merge-sha> HEAD` (exit 0 = present), or compare `git log -1 --format=%ci HEAD` against the PR's merge date. If HEAD predates the change, create the fix branch from fresh `origin/master` first (per [`agent-rules.md`](agent-rules.md) → *Creating a new branch*), then reproduce there. #5528 (a regression from #5504) was invisible for a long stretch because the working branch's base predated #5504 entirely.

## Finding when a code token / heuristic was introduced — `-S` with a filename glob across all history

To trace *when* and *by whom* a specific piece of logic entered the codebase (e.g. "when was this `nextval` identity heuristic added?"), use `git log -S "<token>" --all -- "*<Filename>"` — a **filename glob**, not the full current path. Path-scoped `-S` (and even `git log --follow -S`) miss the lineage when the file later moved namespaces or directories: the path-scoped query returns only the rename commit, `--follow` returns one hop, and the true introduction stays hidden. The `--all -- "*File.cs"` glob form matches the token across every historical location of the file regardless of moves. Once you have the introducing commit, `git show -s --format='%H%n%an%n%ad%n%s' <sha>` gives author + date; note that pre-2011 introductions trace into BLToolkit (see [`predecessor-bltoolkit.md`](predecessor-bltoolkit.md)) and pre-PR-workflow commits (early linq2db history) carry no PR/issue reference at all — a direct push is the honest answer, not a missing link. (Surfaced tracing PR #5647's PostgreSQL `nextval` detection back to a 2013 direct push by the same author who later narrowed it.)

## Behavior-preserving refactor — verify a failing test against the base, don't reason about "pre-existing"

When you believe a refactor preserves behavior and a test then fails, do **not** reason about whether the failure was already there ("the original probably did this too"). Stash your change (or check out PR HEAD) and run the test: the pre-refactor code is the ground truth. When the change under test is a whole PR and its merge-base diverges from current master (`git merge-base origin/master origin/pr/<n>` ≠ `git rev-parse origin/master`), build the comparison worktree at the **merge-base**, not latest master — otherwise unrelated master drift since the PR last merged confounds the red/green attribution (PR #5450's Firebird/Informix/Sybase `UpdateFromSubquery*` regression was confirmed red on PR head / green on the exact merge-base it merged). This bit twice in one session refactoring the YDB scalar-subquery→CTE visitor — `ComparisonTest1` was asserted pre-existing but was green on HEAD (the fold-into-IN-less rewrite regressed it), and reviving a dead `or SqlSetOperator` branch that *looked* like original behavior regressed `CalledWithCorrectNames` (HEAD passed it 1/1). Both were only settled by running the base. Confirm regression-vs-pre-existing empirically — a behavior-exact claim about the original is cheap to test and has been wrong.

## Proving a red test after the fix is committed

When you've already committed the fix and its test together (so the working tree is clean) but want to demonstrate the test is genuinely red without the fix, don't revert the commit or hand-edit core: `git checkout HEAD~1 -- <fix-file>` drops the pre-fix version of *just* that file into the working tree, leaving the (already-committed, or newly-added-and-uncommitted) test intact. Run → confirm red → `git checkout HEAD -- <fix-file>` restores the fix. Verify with `git status` that only the intended file changed before and after. (#5296: used to prove `ProviderDetectionDoesNotLeakConnection` failed `Expected Closed, was Open` on the pre-fix `ProviderDetectorBase` across all 4 TFMs, then restore for the green run. Pairs with [`agent-rules.md`](agent-rules.md) → *A fix in cross-cutting core gets a core-layer regression test*.)

## Triangulate to localize a defect to a code path before reading the implementation

When a failure could live in any of several code paths, localize it empirically before spelunking the source: hold the data fixed and vary **one axis** across a known-good case and the failing case. The narrowest axis that flips pass→fail names the responsible path. On PR #5450 a KeyedQuery eager-load returned empty children for a 9-member composite key; three one-axis variations settled it cheaply — 9-member key under the *Default* strategy worked (so the data + correlation were sound), a *7-member* key under KeyedQuery worked (so KeyedQuery's mechanics were sound), and only the *9-member* KeyedQuery case failed — pinning the bug to KeyedQuery's nested-`ValueTuple` (>7-member) handling, which then made the offending projection obvious. Cheaper and more reliable than reading hundreds of lines first; also distinguishes "two separate defects" from "one" (here a build-time crash *and* a downstream projection bug both keyed off the same >7-member nesting).

## Timing-sensitive (flaky) bug — don't perturb hot paths with per-call I/O

When a bug is order- / timing- / cache-state-sensitive (flips pass↔fail across otherwise-identical runs), instrumentation that does **per-call I/O in a hot path masks it** — the I/O latency changes timing enough to hide the failure. On the #5657 aliasing race, adding `File.AppendAllText` to the `SelectQuery` constructor (hit on every query node) turned the failing test green; the bug only reproduced once the I/O was removed. Instead: stash diagnostic state cheaply on the object (e.g. capture `Environment.StackTrace` into an `internal` field at construction) and write it out **once, at the failure site** (the throw / assertion). That pins the culprit — e.g. the exact `CloneQuery` call-site that created the orphaned node — without altering the timing that produces the bug. Capturing a stack into a field is CPU-only and far less perturbing than file I/O; if even that masks it, narrow the capture to the suspect id range.

## A narrow-code-path repro looks unbuildable — attempt the contrived mapping anyway

When a fix touches a rarely-hit construction path (inheritance + flattened/dotted-`MemberName` columns + calculated columns simultaneously, a specific optimizer branch, etc.), it's tempting to judge a red→green regression test "not practically constructible" and fall back to guarding it only with the existing suite. Attempt the contrived mapping first — linq2db's attribute/fluent mapping is flexible enough to assemble exotic entity shapes that exercise a single code branch, and proving the fix with a test that's **red before / green after** is worth the effort over static reasoning (which is exactly the "too optimistic" failure mode). Iterate the model in `Tests/Tests.Playground/TestTemplate.cs` on SQLite for speed, confirm it goes red on the un-fixed build, then promote it into the proper `Tests/Linq` fixture (restoring the playground scratch). A real example: a calculated column on an inheritance *subtype* whose complex member is mapped via flattened columns reproduced a descriptor-staleness bug that first looked untestable.

## Enabling an `[ActiveIssue]` test after the issue closes

The `/enable-disabled-test` skill drives this end to end; this section is the rationale and the manual fallback.

When a closed issue's `[ActiveIssue]`-gated regression test is being enabled, do **not** trust the close-comment attribution alone (e.g. *"fixed by #X and #Y"*). Bisect to identify the PR that actually flipped the test from fail to pass, and cite that PR in the enable-PR's body. Reporter comments often cite *issues* that share the symptom, not the *PR* that fixed the specific test scenario — the load-bearing fix may be a different PR that landed earlier or later and addressed a broader code path. Concretely:

1. Verify the cited PRs exist (close-comment numbers may be issues, not PRs) and resolve issue→PR via `gh api repos/<o>/<r>/issues/<n>/timeline --jq '.[] | select(.event == "cross-referenced") | .source'` or the GraphQL form in [`pr-resolver.md`](pr-resolver.md).
2. In a worktree (`git worktree add ../<repo>.bisect-<issue> --detach <ref>`), bisect with the test's `[ActiveIssue]` removed locally — typically `git switch --detach <ref>` → edit out the attribute → `dotnet test --project ... --filter <name> -c Debug` → record pass/fail → repeat.
3. Bound the range first (head/master = pass, last release before close = fail), then halve. Each step is a few seconds of build + test.
4. The PR whose merge commit transitions fail → pass is the citation. Mention any reporter-cited related PRs as context, not as the cause.

Bisecting a small range (≤ 50 commits) costs a minute or two and saves re-reviewing a wrong attribution.

## Don't bulk-ungate a multi-cause tracking issue before verifying each case

A long-lived tracking issue (e.g. #5169 "direct vs remote SQL divergence") usually bundles **several unrelated causes across different providers** — its `[ActiveIssue]` / `DisableBaseline` markers are not all the same bug. Before removing a marker, verify *that specific* marker's provider + cause is resolved by your change; don't bulk-remove a family of markers on the hypothesis that one fix covers them all. On #5657 the convert-before-alias fix was assumed to clear the #5169 IN-subquery markers, but removing the ClickHouse+YDB+SqlCe set surfaced that YDB still diverged (alias *numbering* from SourceID-counter state — a different cause), ClickHouse couldn't run locally (no contexts resolved for `--provider ClickHouse.Octonica`), and #5599's SqlCe/YDB markers were still genuinely broken — so all the ungating had to be reverted. Ungate one marker, prove it green on its provider (including the remote `*.LinqService` variant when the marker is about remote/direct parity — see [[feedback_remote_direct_sql_divergence]]), then move to the next.

## Provider SQL limitations — find the flag before gating, then regress wide

When a provider test fails because the generated SQL hits a dialect limitation (e.g. YDB/YQL: comma-cartesian FROM, `JOIN ON` non-equi/constant predicates, aggregate in `ORDER BY`, derived column list on a subquery, `||` on non-strings), **search `SqlProviderFlags` and the `BasicSqlBuilder` virtuals for a flag that already governs that behavior before reaching for an `[ActiveIssue]` / `[DataSources]` gate.** Many provider behaviors are flag-driven and the engine already has the alternate rendering — flipping the flag is a real fix, gating is the fallback. Flags that resolved YDB buckets this way: `IsCrossJoinSyntaxRequired`, `IsSupportsJoinWithoutCondition`, `IsOrderByAggregateFunctionSupported`, `ConcatRequiresExplicitStringCast`, `SupportsColumnAliasesInScalarSource`. **Probe the exact SQL the provider accepts empirically** in `Tests/Tests.Playground/TestTemplate.cs` — run raw dialect variants with `((DataConnection)db).Query<T>(rawSql)` inside `try/catch` and `Assert.Fail(joinedResults)` to dump them (a *passing* NUnit test prints nothing), revert the probe before committing — before deciding fix-vs-gate. **But a minimal probe can under-report the constraint — include the triggering clause.** A stripped repro that omits the clause which actually triggers the limitation gives a false negative: probing `SELECT [p].[X], [d].[X]` (duplicate output names, no `ORDER BY`) wrongly showed SqlCe *accepts* duplicate root column names, but the full suite revealed it errors (`ambiguous column`) only when an `ORDER BY` / name reference resolves against those duplicates. Reproduce the shape the failing query actually generates (ORDER BY, outer wrapping, DISTINCT), or trust a full-suite run over a hand-written snippet. (#5657) Also: an aging provider workaround may be **stale** (YDB moves fast) — re-probe whether it's still needed; but don't *remove* it wholesale, because one flag/workaround often serves several code paths (see next).

**A single flag / builder change can govern multiple code paths, so a targeted fixture baseline-diff will miss collateral regressions.** Setting `SupportsColumnAliasesInSource = false` on YDB correctly dropped the derived column list on a FromSql *scalar subquery* (fixing Issue2779) but **silently broke ~30 VALUES tests** (`Member not found: y.item`) because the same flag also governs the VALUES column list — and the FromSql/Merge baseline-diff didn't include any VALUES fixture, so it looked clean; only the full-suite scan caught it. The fix was to split the flag (`SupportsColumnAliasesInScalarSource` for the subquery path only). Lesson: for a broad provider/`BasicSqlBuilder` flag change, **baseline-diff a full-suite (or every plausibly-affected fixture), not just the obviously-related one** — and when you narrow a workaround, verify the cases it was protecting still pass (removing the YDB `ORDER BY`-by-alias workaround fixed `TakeSkipJoin` but regressed `Distinct`/`Simple4`/`OrderByGroupBy`, which genuinely need the alias under `DISTINCT`/`GROUP BY`).

**Check other-provider precedent before a provider-specific fix / fallback.** Beyond the flags above, grep the sibling `*BulkCopy.cs` / `*SqlBuilder.cs` / `*DataProvider.cs` for how another provider handles the same situation — a match gives a proven shape to mirror: YDB's `BulkUpsert` can't run in a transaction, and `InformixBulkCopy` already gates its provider-specific path on `dataConnection.Transaction == null` and falls back to `MultipleRowsCopy`, so the YDB fix was "do what Informix does." The **absence** of precedent is itself a signal: no provider conditioned a bulk-copy fallback on a *schema* characteristic (a PK column not being supplied) because their bulk APIs auto-generate the key — which flagged that a key-column-based fallback would be bespoke/risky and tipped the decision toward fixing the test instead. Distinguish the kind of precedent you find: a *runtime* condition (Informix's `Transaction == null`) is common; a *schema* condition usually isn't.

## Non-deterministic test results — assert all valid outcomes, don't gate the provider

When a test fails only because its expected value depends on tie-breaking / ordering that no engine guarantees (ties in the sort key, `Take`/`First` over equal values, unspecified row order), prefer rewriting the assertion to accept **all valid results** over excluding/gating the provider. Concretely: `OrderBy(Value1).Take(3).Min(ParentID)` had a tie at `Value1 = 1` (ParentID 1 and 7), so the surviving 3rd row — and thus `Min` — was legitimately `1` or `2` depending on tie-breaking; the in-memory reference used a stable sort (always 1) while YDB picked 7. The fix was `Is.EqualTo(1).Or.EqualTo(2)`, not a `[DataSources(… AllYdb)]` exclusion. Enumerate the possible results first (which rows are *always* in the set, which slot is ambiguous) to bound the valid set. This keeps coverage on every provider and stops the test depending on luck. Gate only when the query genuinely can't run at all.

## Non-deterministic baselines — freeze the source, don't dismiss the drift

A captured baseline `.sql` that changes between regenerations because it embeds a *runtime* value — `NodaTime.SystemClock.Instance.GetCurrentInstant()`, `DateTime.Now`, `Guid.NewGuid()`, random data — is a **test bug**, not reviewable noise. linq2db baselines must be deterministic; a baseline that drifts on every run produces perpetual churn and masks real diffs. Fix it by freezing the source to a fixed value: the test almost always relies only on *relative* arithmetic and row counts, so a constant works without weakening it (e.g. `PostgreSQLTests.Issue5549Populate` used `SystemClock.Instance.GetCurrentInstant()` for the inserted `CreatedAt`; replacing it with a fixed `NodaTime.Instant.FromUtc(2020, 1, 1, 0, 0, 0)` made the captured SQL stable while `now - 7d` / count assertions still held). Reach for `DisableBaseline` only when the value genuinely must be runtime and can't be frozen. This is distinct from the non-deterministic *assertion* case above (ties / ordering) — here the *generated SQL* is non-deterministic, not the expected result. When a PR's baseline diff shows a single-provider drift that is pure timestamp / Guid churn, treat it as a pre-existing test bug to fix, not as drift to ignore.

## Don't fix a failing test by weakening what it exercises

When a provider can't do something a test exercises, don't "fix" it by making the provider silently fall back to a different code path — that hollows out the test (it goes green while no longer verifying its target). Prefer adjusting the test's *setup* so the intended path still runs, or gate it. Example: `BulkCopyDateOnly` (ProviderSpecific) drives YDB's `BulkUpsert`; YDB rejected it because the table's identity key wasn't supplied. A `BulkUpsert → MultipleRowsCopy` fallback would have made it pass but silently stopped testing `BulkUpsert` — so instead the *test table* was changed to use a natural, supplied (non-identity) key, keeping the real path under test. A green test that no longer exercises its target is worse than a red one.

## A regression test's setup must discriminate the fixed path from the buggy path

Choose the column type, mapping, and data so the **buggy** code path and the **fixed** path produce observably different results — otherwise the assertion is vacuous even when it looks strict. Symptom: a test that passes both before and after the fix. Example (#5348): a custom `Contains` translator had to win over the built-in predicate path, but the test column was mapped `DataType.Text`, where *both* paths emit valid SQL returning the same rows — so asserting the rows couldn't catch a dispatch regression. Mapping the column as the faithful `jsonb` type (where the built-in path fails with `like_escape(jsonb, ...)`) made the test load-bearing **and** surfaced a third latent bug (lowering before casting → `lower(jsonb)`, which PostgreSQL has no overload for) that three prior reviewers missed because they all reasoned against the text column. Prefer the faithful repro type over a convenient one: it both discriminates and exposes type-specific bugs the masked setup hides. (Converse of *Don't fix a failing test by weakening what it exercises* above — that rule keeps the intended path under test; this one ensures the assertion can tell the path apart from its bug.)

## Repeated resistance to a correction signals goal misalignment

When the agent (or a subagent) keeps re-doing essentially the same thing after the user has corrected it two or more times, the cause is usually not that the instruction needs to be louder — it's that the agent's *model of the goal* conflicts with the user's. A third near-identical attempt, or restating the correction more forcefully, just repeats the misunderstanding. Stop and surface the conflict instead: state in one line what you believe the goal is and why the current approach serves it, and ask the user to confirm or correct *that*. Often a standing instruction or an earlier assumption is pulling against the new correction; naming it is faster than another failed attempt. (Paired with the interaction-side *two-correction rule* in [`agent-rules.md`](agent-rules.md) → *Batching and user interaction*.)

## "Already-fixed" close-candidates — verify empirically before closing

A static code-archaeology verdict that an issue is "already fixed" (the API/behavior exists in current source, a same-named PR merged) is **unreliable for cross-cutting / translation features** — it routinely overstates. Before closing an issue as fixed:

- **Verify cross-provider coverage, not just that the type/enum/API exists.** The `Int128`/`UInt128`/`Half` enum members exist (#3660), but actual provider support was poor — "the mapping enum is present" ≠ "supported across providers." Check the providers the issue names.
- **Confirm the deeper symptom survived the 6.0 translator rewrite.** A pre-6.0 fix (merged PR with tests) may have regressed in the rewrite. #3560's join-null-equality refactor "landed", but the binary-`+` C#→SQL nullability bug it touched is still live (recent issues filed) — the maintainer knew; static reading didn't.
- **A merged PR that *names* the feature isn't proof the whole issue is resolved** — #3314's ISNULL optimization merged via #3353, but the issue needs both #3353 *and* the still-open WIP #3354 to fully close.
- **Defer to maintainer domain knowledge.** When the maintainer says "not fixed / coverage is poor / needs review," that overrides the static evidence. Present "already-fixed" as a *candidate to confirm*, never an assertion — and only close-as-completed (with the PR/`file:line` citation) once confirmed. This is the empirical-proof side of *never claim a fix works on static reasoning alone* ([`agent-rules.md`](agent-rules.md) → *Before coding a fix or feature*) applied to triage. Pairs with *Don't speculate engine regressions*. Issue typing / close-reasons for this flow live in [`issue-labels.md`](issue-labels.md) → *Triage*.

## A charset/encoding-dependent read/write failure — fix it encoding-independently, don't change the DB/connection encoding

When a provider read or write fails only under a particular database/connection character set (e.g. "Malformed string" reading a binary column over a UTF8 connection), the fix must make the code work under **any** encoding — do **not** "fix" it by switching the connection or database default charset (e.g. to `NONE`). An encoding change masks the gap for one config and breaks another (a `NONE` connection then garbles Unicode text columns), and it's not a fix a real consumer can rely on. The load-bearing example: on Firebird 6 (UTF8-default DB) a `CHAR(16) OCTETS` Guid **read** raises "Malformed string" because the client transliterates the OCTETS bytes to the connection charset; the correct fix reads the value as its canonical text via `UUID_TO_CHAR(...)` in the SELECT projection (charset-independent, applied to all versions since the trigger is the DB encoding, not the server version), **not** a `charset=NONE` connection string. Distinguish this from an issue that is genuinely below linq2db (client/server protocol) — see the next two rules and [[project_5483_fb6_guid_binary_write]] in auto-memory.

## Isolate DDL vs parameter vs read before blaming the value

When a write/round-trip fails at the driver ("Malformed string", "string truncation", a bind error), pin *which* stage fails before theorising a fix. Read the emitted `CREATE TABLE` to confirm the **column type** is what you assumed (a `DataType.Binary(16)` may render as `BINARY(16)` or `CHAR(16) OCTETS` depending on the version builder), then capture the **exact failing SQL statement** (INSERT VALUES vs UPDATE SET vs WHERE compare vs the read) from the trace — don't accept a subagent's summary of "which test/SQL failed" (test-runner can mis-attribute — see [`../agents/test-runner.md`](../agents/test-runner.md) → *Output format* rules). Getting this wrong sends you optimising the read path when the write is at fault, or the parameter binding when the column DDL is. A useful discriminator: an all-zero value that succeeds while a non-zero one fails points at charset validation of the *bytes*, not the SQL shape.

## Raw parameter types (byte[]/string) can dodge provider type-mapping bugs — try, but verify

When a typed/native parameter binding trips a provider bug, binding the raw underlying type (`byte[]`, `string`, the canonical text form) sometimes sidesteps the driver's type mapping and is worth trying. But it is **not** guaranteed and can trade one failure for another: on Firebird 6 neither raw `byte[]`, native `FbDbType.Guid`, an OCTETS parameter charset, nor explicit-OCTETS DDL/CAST got a binary-Guid parameter past the client — all failed identically, proving the issue was below linq2db (FbClient/server), not a binding-shape choice. Try the raw type early, but once several distinct binding shapes fail identically at the same driver frame, stop and conclude it's below the ORM rather than continuing to permute bindings (record it as a dead-end — [[project_5483_fb6_guid_binary_write]]).
