# Per-provider container + DB init for release testing

Release testing (LINQPad 5/7+, T4, NuGet-T4, CLI) reads connection strings from settings directly — providers do **not** need to be enabled in `UserDataProviders.json`. The only env prep is: container start + database init via the matching script under `Data\Setup Scripts\` (in the linq2db repo, not this curation workspace).

Per-provider entries below capture the exact invocation so `release-test-matrix` can run them without re-asking the user.

## Schema

```markdown
## <provider-name>

- **Container:** `<docker-container-name>` (`docker start <name>`)
- **Setup script(s):** `Data\Setup Scripts\<file>.<ext>` <!-- relative to linq2db repo root -->
- **Invocation:** `<exact command line, including pwsh / bash / sqlcmd / docker exec etc>`
- **Env vars:** <comma-separated VAR=VALUE pairs, or "none">
- **Notes:** <quirks: requires admin password, requires container-internal exec, takes ~N seconds, ...>
- **Last verified:** <iso-date> on release <version>
```

## Allowed actions

The release skill may **run** the recorded invocation directly when it's documented here. For undocumented providers, the skill asks the user for the invocation, records it here, and prompts session-reload. The skill does **not** modify or read `Data\Setup Scripts\<file>` itself — those scripts are owned by the linq2db repo's CI/build setup, not by this curation workspace.

## Pollution remediation

Containers accumulate test artifacts over the release cycle: a `CreateDatabase` test from one PR may add a table the canonical baseline doesn't expect, and the table persists across container stops because it lives in the persisted DB volume. When CLI scaffold (track 4.7) or T4 templates (track 4.6 / 4.5) emit diffs for `<Provider>/<TableName>.cs` files that look "wrong" (extra tables, missing tables, FK name churn), the issue is almost always container-side, not the scaffold itself.

Per-provider remediation patterns:

| Symptom | Provider(s) | Remediation |
|---|---|---|
| Extra tables in scaffold output (`Foo`, `TestTable`, `Linqdatatypesbc`, etc.) | Firebird (`firebird25`/`30`/`40`/`50`), MySQL/MariaDB | **Recreate the container** via `Data\Setup Scripts\<provider>.cmd`. The setup script does `docker stop` + `docker rm -f` + `docker run` from a fresh image, which drops the polluted DB volume. After recreate, run `CreateData.CreateDatabase` test for that provider to install the canonical schema. |
| Missing tables in scaffold output (lots of deletions vs committed baseline) | Oracle (`oracle11`/`19`/`23`), any provider whose container was stopped for weeks | **Re-run CreateDatabase** against the right connection name (the `UserDataProviders.json` → `MyConnectionStrings` override may point at a different server than `DataProviders.json` → `LocalConnectionStrings` — the test follows the override). No need to recreate the container — just `dotnet test … --filter "FullyQualifiedName~CreateData.CreateDatabase"` against the provider re-seeds the canonical schema. |
| FK name churn only — same tables, different `[Association(... OtherKey = nameof(<FkName>))]` values on each recreate | Oracle (`SYS_C00xxxx` auto-named constraints), Firebird (`INTEG_xxxx`) | **Known flaky baselines** — these constraint names are auto-generated by the DB on schema-create and shift on every recreate. Treat as noise and restore via `git restore <file>`. The committed baselines hold the original SYS_C/INTEG values from when the baseline was first captured; the scaffold diff reflects new auto-generated names but the schema is structurally identical. Long-term fix would be naming the FKs explicitly in the CreateDatabase test schema. |
| Scaffold produces 0 tables (e.g. `TestDataDB.cs` has no `ITable<>` declarations) | Any provider where the CN points to a server that has the named DB but no tables under the connecting user | The CN in `MyConnectionStrings` points at a different server / user than CreateDatabase populated. Compare `UserDataProviders.json` → `MyConnectionStrings.Connections.<name>.ConnectionString` against the host/user where CreateDatabase actually ran; align them OR run CreateDatabase against the user's actual CN. |

Recorded on 6.3.0 (Firebird needed recreate + CreateDatabase; Oracle needed CN fix in `release-test-cli-scaffold.ps1` to read `MyConnectionStrings` overrides, then CreateDatabase against the real server).

## db2

- **Container:** `db2` (`docker start db2`)
- **Setup script(s):** `Data\Setup Scripts\db2.cmd` (relative to linq2db repo root)
- **Invocation:** `pushd "Data\Setup Scripts" && db2.cmd && popd`
- **Env vars:** none (script uses hardcoded password `Password12!`)
- **Notes:** Container uses Docker-managed volumes (`/database`, `/hadr`). On `docker stop` + `docker start`, the IBM `icr.io/db2_community/db2:latest` image exits with code 1 — the image's `setup_db2_instance.sh` has a chown bug (refs `.fenced` with leading dot; actual file is `fencedid`) AND `db2icrt` rejects the file because it's owned `root:root` instead of `root:db2iadm1` (DBI20187E in db2icrt.log; root cause documented at [IBM community thread](https://community.ibm.com/community/user/discussion/121-container-community-edition-docker-start-fails-dbi20187e)). `db2.cmd` was updated on 6.3.0 to patch the container's setup script in-place via `docker exec sed` immediately after first-run setup completes — inserts a `chown root:db2iadm1 /database/config/db2inst1/sqllib/adm/fencedid` line before the `create_instance` call. Patch persists in the container's writable layer until `docker rm` — re-run `db2.cmd` if the container is recreated.
- **Last verified:** 2026-05-16 on release 6.3.0
