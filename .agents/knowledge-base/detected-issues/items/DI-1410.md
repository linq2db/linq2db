# DI-1410 -- Type-name string matching for EF Core temporal query root dispatch in TransformExpressionVisitor

**Severity**: med  **Category**: code-smell  **Status**: open

## Location
- `Source/LinqToDB.EntityFrameworkCore/Internal/TransformExpressionVisitor.cs:368` -- `string.Equals(expressionTypeName, "TemporalAsOfQueryRootExpression", StringComparison.Ordinal)`
- `Source/LinqToDB.EntityFrameworkCore/Internal/TransformExpressionVisitor.cs:379` -- `string.Equals(expressionTypeName, "TemporalFromToQueryRootExpression", StringComparison.Ordinal)`
- `Source/LinqToDB.EntityFrameworkCore/Internal/TransformExpressionVisitor.cs:392` -- `string.Equals(expressionTypeName, "TemporalBetweenQueryRootExpression", StringComparison.Ordinal)`
- `Source/LinqToDB.EntityFrameworkCore/Internal/TransformExpressionVisitor.cs:405` -- `string.Equals(expressionTypeName, "TemporalContainedInQueryRootExpression", StringComparison.Ordinal)`
- `Source/LinqToDB.EntityFrameworkCore/Internal/TransformExpressionVisitor.cs:418` -- `string.Equals(expressionTypeName, "TemporalAllQueryRootExpression", StringComparison.Ordinal)`

## Pattern matched
`.GetType().Name == "<lit>"` equivalent -- `expressionTypeName` is captured once via `queryRoot.GetType().Name` (line 367) and then compared five times against EF Core internal type-name literals to dispatch SQL Server temporal-table query construction. Same family as DI-0763 (`EFCoreMetadataReader.cs`), which the catalog's type-name-match pattern already covers via `string.Equals(...)`, not just literal `==`.

## Why it matters
EF Core's `TemporalAsOfQueryRootExpression`/`TemporalFromToQueryRootExpression`/etc. live in `Microsoft.EntityFrameworkCore.Query.Internal`, so there is no version-stable compile-time type reference across the four EF targets this file compiles against (EF31/8/9/10). If EF Core renames or restructures one of these types in a future release -- as already happened once with Npgsql's `PgBinaryExpression`, see DI-0763 -- the matching `string.Equals` check silently fails and `TransformQueryRootExpression` falls through to `return getTableExpr`: the SQL Server temporal-table hint (`AsOf`/`FromTo`/`Between`/`ContainedIn`/`All`) silently stops being applied instead of throwing, so a query using `.TemporalAsOf(...)` still runs, but against the *current* table state instead of the requested point in time. That is a silent correctness regression, not a build break, so it would only surface via a user-reported wrong-result bug.

## Suggested fix
Where the concrete EF Core type is `public` (most temporal query-root types are, despite living in an `.Internal` namespace), reference it directly with a type pattern (`queryRoot is TemporalAsOfQueryRootExpression asOf`) guarded by the existing `#if !EF31` symbol, keeping the string-based dispatch only where the type is genuinely inaccessible for a given EF target. At minimum, hoist the five literals into named `const string` fields next to `PgBinaryExpressionName` in `EFCoreMetadataReader.cs`, following the same "renamed in X.0" comment convention, so a future EF rename is a one-line fix instead of a five-site hunt.
