# DI-1480 -- PostgreSQL 19 CI containers pin a pre-release beta Docker tag

**Severity**: med  **Category**: tech-debt  **Status**: open

## Location
- `Build/Azure/scripts/pgsql19.sh:3` -- `docker run` pins `postgres:19beta1`
- `Build/Azure/scripts/mac.pgsql19.sh:3` -- `docker run` pins `postgres:19beta1`

## Pattern matched
Build/CI hardcoded-value sweep (delta task supplement to the code pattern catalog): both new scripts hardcode `postgres:19beta1` as the container image tag. Every other PostgreSQL CI script in the same directory (`pgsql13.sh` through `pgsql18.sh`, and their `mac.*` counterparts) pins a plain GA major-version tag (`postgres:13` ... `postgres:18`) -- confirmed by diffing `pgsql17.sh`/`pgsql18.sh`/`mac.pgsql17.sh`/`mac.pgsql18.sh` against these two files. The beta tag is a deliberate, but time-limited, deviation from that convention.

## Why it matters
Docker Hub routinely stops publishing prerelease tags once the corresponding GA image ships, so `postgres:19beta1` can silently start failing to pull with no linq2db code change to explain the new CI failure (or, on a runner with a stale local image cache, silently keep testing against an outdated beta build instead of GA). The BUILD area's own KB notes already anticipate this ("expect a tag swap once PostgreSQL 19 reaches GA"), but nothing in the repo tracks it as an actionable follow-up -- no tracking issue, no inline TODO.

## Suggested fix
Once PostgreSQL 19 reaches GA, swap both `postgres:19beta1` references to `postgres:19` (matching the `pgsql17.sh`/`pgsql18.sh` convention). Consider opening a tracking GitHub issue, or leaving a `# TODO` comment next to both `docker run` lines, so the follow-up surfaces via a future `kb-issue-detector` sweep or a plain `grep TODO` instead of relying on someone remembering this KB note.
