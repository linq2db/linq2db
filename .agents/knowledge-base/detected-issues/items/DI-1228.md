# DI-1228 -- AccessMemberTranslator TranslateRoundToEven tie-break predicate compares against the wrong literal (dead branch)

**Severity**: med  **Category**: dead-code  **Status**: open

## Location
- `Source/LinqToDB/Internal/DataProvider/Access/Translation/AccessMemberTranslator.cs:239`

## Pattern matched
Structural review of `MathMemberTranslator.TranslateRoundToEven` (precision-0 banker's-rounding branch). The method's own leading comment documents the intended tie-break predicate as `Int([Value]) Mod 2 = 0`, but the generated expression at line 239 is `factory.Equal(factory.Mod(intCast, factory.Value(2)), factory.Value(2))` -- comparing an `x Mod 2` result against the literal `2`, a value `Mod 2` can never produce (only `0` or `1`). The predicate is therefore always false, and the `SearchCondition` built from `is5 AND isEven` can never be satisfied, so the true branch (`intCast`, the round-down value) is unreachable in both the C# translator and the SQL it emits -- every call falls through to the `ROUND(value)` false branch. This mismatch between the documented intent and the shipped literal was first surfaced by the area's architecture indexer (see `areas/PROV-ACCESS/INDEX.md` -> Known issues / debt, item 8) and explicitly flagged there for `kb-issue-detector` triage; it is formalized here as this run's delta scan covers the same file.

## Why it matters
The generated SQL always evaluates `Int(v) Mod 2 = 2`, which is impossible, so the round-half-to-even tie-break this code was written to implement never actually fires -- behavior for a `.5` tie at precision 0 depends entirely on Access's native `ROUND` function agreeing with .NET's `MidpointRounding.ToEven` semantics, which is undocumented and unverified in this codebase. Because the adjacent comment states the intended predicate correctly, a future maintainer skimming the comment (rather than the generated expression tree) could believe the tie-break branch is live and build on that assumption, or "helpfully" touch nearby code without noticing the branch never executes.

## Suggested fix
Change the literal on line 239 from `factory.Value(2)` to `factory.Value(0)`, restoring the `Int([Value]) Mod 2 = 0` predicate the comment already documents. Before landing, add a regression test that drives a `Math.Round(x, MidpointRounding.ToEven)` call at precision 0 with a `.5` tie (including a negative-value tie) against Access, since the area's own notes suggest native `ROUND` may already cover this case -- confirm the fix does not change observable output before removing the "likely benign" caveat from the area index.
