# DI-1231 -- YdbProviderAdapter blocks a thread on async DescribeTable via GetAwaiter().GetResult()

**Severity**: med  **Category**: perf-smell  **Status**: open

## Location
- `Source/LinqToDB/Internal/DataProvider/Ydb/YdbProviderAdapter.cs:145` -- primary-key lookup lambda wired into `_describePrimaryKey`

## Pattern matched
`\.GetAwaiter\(\)\.GetResult\(\)` in non-test code (async-over-sync / perf-smell pattern from the catalog). The mapped lambda calls the SDK's async `YdbSchema.DescribeTable(...)` and immediately blocks the calling thread with `.GetAwaiter().GetResult()` to force a synchronous result.

## Why it matters
`SchemaProviderBase.GetPrimaryKeys` (and the whole `GetSchema` pipeline it belongs to) is sync-only -- there is no async counterpart for `YdbProviderAdapter` to call instead. Blocking on `.GetAwaiter().GetResult()` ties up a thread-pool thread for the duration of the gRPC round-trip and is deadlock-prone if schema introspection ever runs under a captured `SynchronizationContext` (e.g. scaffolding tooling hosted inside a UI or classic ASP.NET pipeline). It's low-frequency (schema/scaffolding path, not the hot query path), but the pattern would resurface for any future provider wiring a similarly async-only SDK call into this sync-only base contract.

## Suggested fix
No local one-line fix: the root cause is that `SchemaProviderBase` exposes no async `GetPrimaryKeys`/`GetSchemaAsync` surface, so any provider backed by an async-only client is forced into this shape. A real fix means adding an async overload to `SchemaProviderBase` -- a cross-cutting change affecting every provider's schema provider, out of scope for a provider-local change. Until such a seam exists, this stays a documented, accepted tradeoff rather than something fixable in `YdbProviderAdapter.cs` alone.
