# DI-1320 -- Query<T> constructor made public without an XML doc summary

**Severity**: low  **Category**: doc-gap  **Status**: open

## Location
- `Source/LinqToDB/Internal/Linq/Query{T}.cs:20` -- `public Query(IDataContext dataContext)`

## Pattern matched
"Public API method without XML doc (when file is Tier 1 of any area)" -- `Query{T}.cs` is a pinned Tier-1 file for the LINQ area. The constructor's visibility was widened from `internal` to `public` (part of the QueryCache refactor -- "so tests can construct stub Query<T> instances") but no `<summary>` / `<param>` doc was added, leaving a bare public entry point on a sealed public type undocumented.

## Why it matters
`Query<T>` is now constructible from outside the assembly with no guidance on when a caller should do so, what `dataContext` must supply, or that the resulting instance is otherwise built exclusively through `Query<T>.GetQuery` / `CreateQuery` and the translator pipeline. AGENTS.md requires XML docs on new public members; an undocumented public ctor on a widely-referenced internal-namespace type is exactly the kind of surface a consumer can misuse (e.g. constructing a `Query<T>` and expecting it to be immediately executable) without a doc comment to correct that assumption.

## Suggested fix
Add a `/// <summary>` (plus `<param name="dataContext">`) on the constructor clarifying it produces an uninitialized `Query<T>` shell intended for the query-build pipeline (`ExpressionBuilder`) or test stubbing, not a ready-to-execute query -- mirroring the doc style already present on `ClearCache()` / `CacheMissCount` a few lines down in the same file.
