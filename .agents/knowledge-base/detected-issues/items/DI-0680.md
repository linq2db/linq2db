# DI-0680 — Empty catch blocks in Drop/DropAsync swallow all exceptions when throwExceptionIfNotExists=false

**Severity**: med  **Category**: code-smell  **Status**: open

## Location
- `Source/LinqToDB/LinqExtensions/LinqExtensions.cs:291` — `catch when (!throwExceptionIfNotExists) { }` in `Drop<T>`
- `Source/LinqToDB/LinqExtensions/LinqExtensions.cs:328` — `catch when (!throwExceptionIfNotExists) { }` in `DropAsync<T>`

## Pattern matched
Catch block with empty body (catalog pattern: "Catch block with empty body or only `// ignore` comment"). Both the sync `Drop` and async `DropAsync` overloads contain an empty catch-when block that swallows every exception type — including transient infrastructure errors, permission failures, and network timeouts — whenever `throwExceptionIfNotExists` is `false`.

## Why it matters
The doc comment for the parameter explicitly acknowledges: "This behavior is not correct and will be fixed in future to mask only missing table exceptions." Using `throwExceptionIfNotExists=false` as a fire-and-forget guard will silently suppress connection drops, deadlocks, or access-denied errors, making those failures invisible to callers. Any caller relying on this flag for "safe drop" will unknowingly hide real failures.

## Suggested fix
Replace the catch-all with a targeted filter: catch only the provider-specific "table does not exist" exception (e.g. check `SqlException.Number == 3701` for SQL Server, or catch `LinqToDBException` after inspecting the message). A `ProviderSpecificTableNotFoundException` or a `SqlProviderFlags.SupportsDropTableIfExists` guard emitting `DROP TABLE IF EXISTS` (where supported) would be the cleanest resolution and eliminate the need for exception-based control flow entirely.
