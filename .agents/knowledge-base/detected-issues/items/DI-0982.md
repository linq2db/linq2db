# DI-0982 -- SQLiteProviderDetector has redundant hardcoded literal alongside its constant

**Severity**: low  **Category**: code-smell  **Status**: open

## Location
- `Source/LinqToDB/Internal/DataProvider/SQLite/SQLiteProviderDetector.cs:25` -- case "Microsoft.Data.SQLite" in DetectProvider(ConnectionOptions) switch
- `Source/LinqToDB/Internal/DataProvider/SQLite/SQLiteProviderDetector.cs:81` -- case "Microsoft.Data.SQLite" in DetectProvider(ConnectionOptions, SQLiteProvider) switch

## Pattern matched
Hardcoded provider-name string literal in switch/case alongside an existing constant. The adjacent case arm at line 24 uses SQLiteProviderAdapter.MicrosoftDataSQLiteClientNamespace for the same provider namespace. The bare literal "Microsoft.Data.SQLite" appears in both DetectProvider overloads. If the constant and the literal are equal the literal is dead code; if they differ this is a silent detection split.

## Why it matters
If SQLiteProviderAdapter.MicrosoftDataSQLiteClientNamespace is ever changed (e.g. casing correction to Microsoft.Data.Sqlite), the literal case silently diverges and causes a detection miss or unreachable branch. Maintainers cannot tell from reading the switch whether the literal is intentionally distinct from the constant.

## Suggested fix
Verify whether SQLiteProviderAdapter.MicrosoftDataSQLiteClientNamespace equals "Microsoft.Data.SQLite" at runtime. If equal, remove the duplicate literal case. If different, define a named constant and replace the literal.
