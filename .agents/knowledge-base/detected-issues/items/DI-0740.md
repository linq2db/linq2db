# DI-0740 — SqlProviderFlags.IsDerivedTableOrderBySupported has copy-paste XML doc

**Severity**: low  **Category**: doc-gap  **Status**: open

## Location
- `Source/LinqToDB/Internal/SqlProvider/SqlProviderFlags.cs:482` — `IsDerivedTableOrderBySupported` property

## Pattern matched
Public API method without correct `<summary>` XML doc (Tier-1 file). The `<summary>` says "Provider supports SUM/AVG/MIN/MAX(DISTINCT column) function. Otherwise, it will be emulated." — which describes a different flag — copied from a neighboring property instead of describing derived-table ORDER BY support.

## Why it matters
Any tooling or developer consulting IntelliSense / XML doc for `IsDerivedTableOrderBySupported` will read a description that belongs to a different capability. New provider implementors are likely to misunderstand what the flag controls, potentially leaving it at its default `false` when their provider does support derived-table ORDER BY (or vice-versa). The mismatch is invisible at compile time because the property name and type are still correct.

## Suggested fix
Replace the `<summary>` with the accurate description, e.g.:
- "Provider supports ORDER BY in derived tables (subqueries used as table sources). Default (set by `DataProviderBase`): `false`."
