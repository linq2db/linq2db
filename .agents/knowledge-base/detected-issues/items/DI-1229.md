# DI-1229 — GetMappingSchema switch has no v13 arm; PostgreSQL13MappingSchema is never selected

**Severity**: high  **Category**: code-smell  **Status**: open

## Location
- `Source/LinqToDB/Internal/DataProvider/PostgreSQL/PostgreSQLDataProvider.cs:602-613`

## Pattern matched
Structural dispatch-completeness check (code-smell — omission in a switch expression), same shape as the already-tracked DI-0960 float-precision bug in this same file. `PostgreSQLDataProvider.GetMappingSchema` switches on `PostgreSQLVersion` with explicit arms for `v19`, `v18`, `v15`, `v92`, `v93` and a `_` default — there is no `PostgreSQLVersion.v13` arm, so a v13 provider falls through to the default and receives `PostgreSQLMappingSchema.PostgreSQL95MappingSchema` instead of the dedicated `PostgreSQLMappingSchema.PostgreSQL13MappingSchema` sealed class defined in `PostgreSQLMappingSchema.cs:208`. Every other per-version dispatch point in this same file (`GetProviderName`, `CreateMemberTranslator`, `CreateSqlBuilder`) and in `PostgreSQLProviderDetector.cs` (`DetectProvider`, `GetDataProvider`, `DetectServerVersion`) has a v13-handling branch — `GetMappingSchema` is the outlier.

## Why it matters
`PostgreSQL13MappingSchema` and `PostgreSQL95MappingSchema` share identical converter/scalar-type content (both derive everything from the shared `Instance`); the only functional difference between the two sealed classes is the `ProviderName` passed to their `LockedMappingSchema` base constructor, which is what a v13-configured connection's `MappingSchema.ConfigurationList` reports. Any code that branches on `ConfigurationList.Contains(ProviderName.PostgreSQL13, ...)` gets a false negative for such a connection. This already exists in this exact codebase: `PostgreSQLHints.cs`'s `SubQueryTableHintExtensionBuilder` SKIP LOCKED check ORs together `PostgreSQL95`, `PostgreSQL13`, `PostgreSQL15`, `PostgreSQL18`, `PostgreSQL19` — it still passes for a v13 connection only because `PostgreSQL95` is also in that OR chain. A future capability check that tests `PostgreSQL13` alone (without the same coincidental `PostgreSQL95` fallback) would silently misbehave for v13-configured connections.

## Suggested fix
Add a `PostgreSQLVersion.v13 => new PostgreSQLMappingSchema.PostgreSQL13MappingSchema(),` arm to the switch in `GetMappingSchema`, mirroring the v13 arms already present in `GetProviderName`/`CreateMemberTranslator`/`CreateSqlBuilder` in the same file.
