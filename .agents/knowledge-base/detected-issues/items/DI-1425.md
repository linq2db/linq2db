# DI-1425 — YdbProvider.ClearAllPools blocks a thread on async YdbConnection.ClearAllPools via GetAwaiter().GetResult()

**Severity**: med  **Category**: perf-smell  **Status**: open

## Location
- `Source/LinqToDB.LINQPad/DatabaseProviders/YdbProvider.cs:24` — `ClearAllPools` override

## Pattern matched
`\.GetAwaiter\(\)\.GetResult\(\)` in non-test code (async-over-sync / perf-smell pattern from the catalog). `YdbConnection.ClearAllPools()` is `Ydb.Sdk`'s async pool-teardown API; the override blocks the calling thread with `.GetAwaiter().GetResult()` to satisfy the synchronous `IDatabaseProvider.ClearAllPools(string)` contract.

## Why it matters
`ClearAllPools` is invoked from the LINQPad driver's connection-teardown/refresh path, which in `LinqToDBDriver`/`LinqToDBStaticDriver` typically runs on the UI thread (schema refresh, "Clear all pools" action, connection-dialog retry). Blocking on `.GetAwaiter().GetResult()` there risks a UI freeze for the duration of the gRPC pool-close round-trip, and is deadlock-prone if ever invoked under a captured `SynchronizationContext`. Same shape as the already-tracked `YdbProviderAdapter` finding (DI-1231) in PROV-YDB — this is the second instance of the sync-over-async tradeoff introduced by wiring the async-only `Ydb.Sdk` client into a synchronous contract, this time in the LINQPad companion driver.

## Suggested fix
No local one-line fix: `IDatabaseProvider.ClearAllPools(string providerName)` is synchronous across all 15 concrete providers (`DatabaseProviderBase.ClearAllPools` is `abstract void`), so `YdbProvider` has no async surface to call into instead. A real fix would add an async `ClearAllPoolsAsync` to `IDatabaseProvider`/`DatabaseProviderBase` and thread it through both LINQPad drivers' teardown call sites — a cross-cutting interface change out of scope for this file alone. Until that seam exists, this is a documented, accepted tradeoff rather than something fixable locally.
