# DI-1380 -- TypeMapper's public API surface lacks XML doc summaries

**Severity**: low  **Category**: doc-gap  **Status**: open

## Pattern matched
"Public API method without <summary> XML doc (when file is Tier 1 of any area)". TypeMapper.cs is pinned as a Tier-1 file for INTERNAL-API in kb-areas.md. The class itself carries a one-line summary ("Implements typed mappings support for dynamically loaded types."), but none of its ~35 public members have one -- RegisterTypeWrapper, FinalizeMappings, the MapExpression/MapAction/MapLambda/MapActionLambda/BuildFunc/BuildAction overload families, Type<T>(), the BuildWrappedFactory/BuildFactory/BuildTypedFactory families, Wrap<TR>, WrapTask<TR>, and the nested MemberBuilder<T,TV>/TypeBuilder<T> classes are all undocumented.

## Why it matters
TypeMapper is the reflection-bridging layer providers use to wrap dynamically-loaded ADO.NET types. A provider author building a `TypeMapper.Type<T>().Member(...)` chain, or calling RegisterTypeWrapper/FinalizeMappings directly, gets no in-IDE guidance on argument shape, ordering constraints (RegisterTypeWrapper throws once FinalizeMappings has run), or which of the six MapExpression/MapAction arities to pick. Purely a documentation gap -- it doesn't affect runtime correctness -- but it raises the bar for anyone extending provider support against this type.

## Suggested fix
Add `<summary>` (and `<param>`/`<returns>` where the shape isn't obvious) to the public members listed above. Prioritize the ones with non-obvious preconditions first: RegisterTypeWrapper's post-FinalizeMappings throw, and Wrap<TR>'s dependency on a previously-registered wrapper factory.
