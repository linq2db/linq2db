## Why it matters
`SqlField.ColumnDescriptor` is set unconditionally (`= null!`), including on synthetic/non-column `SqlField` instances (e.g. `SqlField.All`, fake fields) where no real column descriptor applies. Code that reads `ColumnDescriptor` without checking for a non-column field can silently pick up wrong metadata (system type, DB type, nullability) instead of failing fast, and the `null!` bypass hides the gap from the nullable-reference-type checker.

## Suggested fix
Introduce a dedicated non-column-field representation, or make `ColumnDescriptor` genuinely nullable without the `null!` bypass, so callers can distinguish "real column, has descriptor" from "synthetic field, no descriptor" at the type level instead of relying on convention.
