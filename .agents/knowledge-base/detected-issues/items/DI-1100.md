# DI-1100 — Type-name string matching for provider detection in LinqToDBForEFToolsImplDefault

**Severity**: med  **Category**: code-smell  **Status**: open

## Location
- `Source/LinqToDB.EntityFrameworkCore/LinqToDBForEFToolsImplDefault.cs:253` — `connection.GetType().Name switch` for DbConnection type
- `Source/LinqToDB.EntityFrameworkCore/LinqToDBForEFToolsImplDefault.cs:274` — `extensions.GetType().Name switch` for RelationalOptionsExtension type
- `Source/LinqToDB.EntityFrameworkCore/LinqToDBForEFToolsImplDefault.cs:409` — `string.Equals(mapping.GetType().Name, "NpgsqlEnumTypeMapping", ...)` for NpgsqlEnumTypeMapping detection

## Pattern matched
`\.GetType\(\)\.Name` compared to string literals — the catalog's type-name string match pattern (`code-smell`, `low`) — elevated here because provider detection by string-name is a known fragility vector: the same pattern in `EFCoreMetadataReader.cs` is already tracked as DI-0763 (triaged, `med`).

## Why it matters
String-based type-name dispatch breaks silently when a provider renames its connection / options-extension class (e.g. a major EF provider upgrade), when a type is moved to an internal assembly, or when a new provider ships a subclass. The switch falls through to the `_` arm and throws `LinqToDBForEFToolsException` with a misleading "Can not detect data provider" message even though the user has a valid, supported provider. The `NpgsqlEnumTypeMapping` check at line 409 is especially fragile: it gates PostgreSQL enum-to-SQL-literal mapping on a type name from Npgsql that has already changed across major versions.

## Suggested fix
- For connection/extension dispatch: use `is` type pattern matching (`connection is NpgsqlConnection`) or a dictionary keyed on `Type` (registered at startup). EF Core itself uses `IDbContextOptionsExtension` polymorphism for the extensions case — prefer `options.Extensions.OfType<NpgsqlOptionsExtension>().Any()` over `.Name ==`.
- For `NpgsqlEnumTypeMapping`: check for a known interface or base class from `Npgsql.EntityFrameworkCore.PostgreSQL`; if none is available, document the version dependency in a comment and add a version guard.
