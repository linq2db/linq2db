## Why it matters
The comment flags a structural ambiguity in `SqlOutputClause`: `HasOutputItems` is a boolean guard used separately from `OutputItems`, requiring callers to coordinate both fields manually. The inline `null!` suppression signals the NRT contract is not enforced here. If this coupling is mishandled in a future visitor pass, a `NullReferenceException` or incorrect clone (sharing the original list instead of copying) could silently corrupt the output-clause AST during query transformation.

## Suggested fix
Encapsulate the `HasOutputItems` / `OutputItems` invariant inside `SqlOutputClause` — either a single nullable property that expresses both pieces of state, or a factory method that ensures consistency — so visitor code does not need to replicate the guard.
