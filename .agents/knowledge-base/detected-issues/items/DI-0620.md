# DI-0620 — Binary compatibility shim: public class and all public members lack XML doc

**Severity**: low  **Category**: doc-gap  **Status**: open

## Location
- `Source/LinqToDB/Compatibility/System/Data/Linq/Binary.cs:13` — public sealed class Binary declaration with no `<summary>`

## Pattern matched
"Public API method without `<summary>` XML doc (when file is Tier 1 of any area)" — `Binary` is a public top-level type in `Source/LinqToDB/` not under `LinqToDB.Internal.*`, making this file Tier-1. None of the public members (`Binary(byte[]?)`, `ToArray()`, `Length`, `implicit operator Binary`, `Equals(Binary?)`, `operator ==`, `operator !=`, `override Equals(object?)`, `GetHashCode()`, `ToString()`) carry a `<summary>` XML doc comment. Only the private `ComputeHash()` has an inline comment.

## Why it matters
This compatibility shim re-exports `System.Data.Linq.Binary` for non-netfx targets. Users who discover this type via IntelliSense or API browsers get no documentation. The class is intentionally part of the public API surface (no `[EditorBrowsable(EditorBrowsableState.Never)]`), so discoverability matters. Without XML docs, tooling-generated reference docs for linq2db will silently omit descriptions for this type's members, and consumers filling in the constructor parameter `value` have no hint about null-handling semantics (null becomes an empty array).

## Suggested fix
Add `<summary>` docs to the class and its public members. Minimum: class summary explaining this is a LINQ-to-SQL compatibility shim for platforms that don't ship `System.Data.Linq.Binary`. Constructor: document null-coalesces to empty array. `ToArray()`: returns a copy. `Length`: byte count of internal array.
