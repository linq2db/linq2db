# DI-0733 -- Public methods on RemoteDataContextBase lack XML doc summaries

**Severity**: low  **Category**: doc-gap  **Status**: open

## Location
- `Source/LinqToDB/Remote/RemoteDataContextBase.cs:55` -- `AddMappingSchema`
- `Source/LinqToDB/Remote/RemoteDataContextBase.cs:544` -- `BeginBatch`
- `Source/LinqToDB/Remote/RemoteDataContextBase.cs:553` -- `CommitBatch`
- `Source/LinqToDB/Remote/RemoteDataContextBase.cs:555` -- `CommitBatchAsync`
- `Source/LinqToDB/Remote/RemoteDataContextBase.cs:632` -- `Dispose` (override)
- `Source/LinqToDB/Remote/RemoteDataContextBase.cs:639` -- `DisposeAsync` (override)

## Pattern matched
"Public API method without `<summary>` XML doc (when file is Tier 1 of any area)". `RemoteDataContextBase.cs` is Tier 1 for REMOTE-CLIENT. All six members are public on an abstract base class decorated with `[PublicAPI]`, and none carries a `<summary>` element. `ConfigureAsync` and `UseOptions`/`UseMappingSchema` are correctly documented via `<summary>` or `<inheritdoc>`.

## Why it matters
`RemoteDataContextBase` is the integration surface that consumer projects subclass to build custom remoting transports. Undocumented public members force implementers to read the source -- particularly `BeginBatch`/`CommitBatch`, whose relationship (must call `BeginBatch` before `CommitBatch`, nested calls stack the counter) is non-obvious from the signature. `AddMappingSchema` has subtly different semantics from `IDataContext.SetMappingSchema` (prepend vs. replace) and that distinction is invisible without docs.

## Suggested fix
Add `<summary>` blocks to each member. For `Dispose`/`DisposeAsync` an `<inheritdoc/>` is sufficient. For batch members, document the counter semantics. For `AddMappingSchema`, note that the schema is prepended (higher priority), unlike `SetMappingSchema` which replaces.
