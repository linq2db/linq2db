## Pattern matched
Doc-gap catalog entry: public API method without a summary XML doc, when the file is Tier 1 of any area. WindowFunctionsMemberTranslator lives in namespace LinqToDB.Linq.Translation, not Internal.*, and the type is public, so per the coverage-tiers Public-API-surface rule the file is Tier 1 regardless of its absence from the area's named Tier-1 anchor list. 13 of the 44 methods above predate this diff (pre-existing gap, never previously indexed); the Window-Functions PR merged in this delta added the remaining methods (percentile, hypothetical-set, statistical, regression, navigation functions) without XML docs either, so the gap grew substantially in this change.

## Why it matters
This class is the provider-subclassing point for window and statistical-function SQL translation -- every provider area overrides a subset of these public virtual methods to adapt syntax (for example Oracle's native RATIO_TO_REPORT, or ClickHouse's *_POP / *_SAMP-only variance family). An undocumented public virtual surface forces both in-tree and third-party provider authors to read the base implementation to learn a method's contract (which arguments are optional, null-handling, when it is invoked) instead of getting it from IntelliSense or the generated API reference, and API-doc generation emits nothing for these 44 public members.

## Suggested fix
Add a one-line summary (and param/returns tags where the emitted SQL or override contract is non-obvious) to each public virtual Translate* method, following the documentation style already used on the public IUpsertSpec / IEntityInsertSpec family introduced in the same PR batch. Several methods already carry a plain comment immediately above the signature (for example on TranslateRatioToReport and TranslateMedian) that can be promoted into the summary directly.
