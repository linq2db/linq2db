# DI-1361 -- CTE field sync keeps a name/index-based fallback alongside the new direct-reference design

**Severity**: med  **Category**: tech-debt  **Status**: open

## Location
- `Source/LinqToDB/Internal/SqlQuery/SqlCteTable.cs:123` -- `GetKeys` falls back to name-based field lookup when the direct `CteField` reference doesn't resolve
- `Source/LinqToDB/Internal/SqlQuery/Visitors/SqlQueryColumnOptimizerVisitor.cs:447` -- `SynchronizeCteFields` falls back to positional-index matching when `SqlCteField.Column` is unset

## Pattern matched
Structural tech-debt (no regex in the catalog covers this shape): this delta introduces a reference-based redesign for CTE field tracking (`SqlCteField.Column` / `SqlCteTableField.CteField`, replacing the old name-string matching), but both call sites that consume it retain the pre-redesign fallback path (name-string equality / positional index) rather than requiring the direct reference. The area's own INDEX.md documents this explicitly: "CTE field lookup has two fallback paths ... the fallback exists for construction paths that don't yet populate the direct reference. A v7 cleanup could require the direct reference unconditionally and drop the string fallback."

## Why it matters
Two independent matching strategies for the same relationship (direct object reference vs. name/index) can silently diverge: a construction path that populates the reference incorrectly (or not at all) degrades to the weaker fallback instead of failing, and any future change to one fallback's semantics needs mirroring in the other by convention, not by the type system. One more construction path that needs CTE field resolution (e.g. a new provider-specific CTE emission, or a new optimizer pass) is likely to copy whichever fallback it finds nearby, growing the divergence surface instead of shrinking it.

## Suggested fix
Audit which construction paths still leave `SqlCteField.Column` / `SqlCteTableField.CteField` unset, populate the reference at construction time for all of them, then drop both string/index fallbacks and assert (or throw) on an unresolved reference instead of silently degrading -- turning a latent correctness gap into a build/test-time failure.
