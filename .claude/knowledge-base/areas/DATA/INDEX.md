# DATA

## What this area does

`Source/LinqToDB/Data/**` is the **public `LinqToDB.Data` namespace** — the persistent-connection façade (`DataConnection`), raw-SQL command surface (`CommandInfo` + `DataContextExtensions.Query`/`Execute`/`ExecuteReader` overloads), bulk-copy entry points (`DataContextExtensions.BulkCopy`/`BulkCopyAsync`), tracing primitives (`TraceInfo`, `TraceInfoStep`, `TraceOperation`), the retry-policy framework (`IRetryPolicy`, `RetryPolicyBase`, `RetryingDbConnection/Command`), and the option records that feed `DataOptions` from this layer (`ConnectionOptions`, `BulkCopyOptions`, `QueryTraceOptions`, `RetryPolicyOptions`). Compared to [CORE](../CORE/INDEX.md), DATA is the *ADO.NET execution site*: it owns `DbConnection`/`DbCommand` lifecycle, parameter binding, reader materialization, transactions, and the `IQueryRunner` implementation that hands prepared SQL off to the ADO provider. AST construction, builder dispatch, and SQL emission are out of scope — see [SQL-AST](../SQL-AST/INDEX.md), [EXPR-TRANS](../EXPR-TRANS/INDEX.md), [SQL-PROVIDER](../SQL-PROVIDER/INDEX.md).

## Subsystems

- **Persistent connection (`DataConnection` partial class).** Six partial files compose one type. `DataConnection.cs` carries ctors (16 obsoleted overloads forwarding to `DataConnection(DataOptions)`, `Source/LinqToDB/Data/DataConnection.cs:39–501`), `DbConnection` lifecycle (`OpenDbConnection`/`OpenConnection`/`Close`, `:790–927`), command lifecycle (`InitCommand`/`CommitCommandInit`/`DisposeCommand`, `:968–1072`), the synchronous `ExecuteNonQuery`/`ExecuteScalar`/`ExecuteReader` triplet wrapped by interceptor + tracing (`:1076–1454`), transaction methods (`:1495–1659`), `MappingSchema` slot (`:1722–1779`), `IDisposable` (`:1801`), and the `UseOptions`/`UseMappingSchema` scoped-override pair (`:1824–1866`). `DataConnection.Async.cs` mirrors every execution path with `ValueTask`/`Task` returns and `ConfigureAwait(false)` discipline (`Source/LinqToDB/Data/DataConnection.Async.cs:54–777`). `DataConnection.Configuration.cs` hosts the static config registry (`ConfigurationInfo`, `_dataProviders`, `AddProviderDetector`, `Source/LinqToDB/Data/DataConnection.Configuration.cs:18–456`) plus the static ctor that chains every embedded provider's `ProviderDetector.DetectProvider` (`:194–211`), and the nested `ConfigurationApplier` static class with `Apply`/`Reapply` for each option set (`ConnectionOptions`/`RetryPolicyOptions`/`DataContextOptions`/`QueryTraceOptions`, `:502–854`). `DataConnection.Interceptors.cs` declares the `IInterceptable<…>` partial implementations for all eight interceptor kinds (`Source/LinqToDB/Data/DataConnection.Interceptors.cs:8–25`). `DataConnection.Linq.cs` wires `IDataContext.SqlProviderFlags`/`DataReaderType`/`CreateSqlBuilder`/`GetSqlOptimizer` through to the `DataProvider` (`Source/LinqToDB/Data/DataConnection.Linq.cs:21–43`).
- **Query runner (`DataConnection.QueryRunner`).** Nested `internal sealed class QueryRunner : QueryRunnerBase` is the `IQueryRunner` implementation `IDataContext.GetQueryRunner` returns for any LINQ query executed against a `DataConnection` (`Source/LinqToDB/Data/DataConnection.QueryRunner.cs:24–34`). It builds a `PreparedQuery` (per-statement command-list + `SqlStatement`) by running the optimizer/convert visitors, applying aliases, and rendering each command via `sqlBuilder.BuildSql` (`:180–292`). Parameter binding goes through `CreateParameter` which reconciles `DbDataType` against `MappingSchema` and lets the `IDataProvider` set provider-specific parameter shape (`:323–347`). Multi-command statements (e.g. an INSERT followed by `SELECT @@IDENTITY`, or a DROP-then-CREATE batch) are handled by per-index `InitCommand` calls and a "swallow exceptions on intermediate `DROP`" rule (`:419–443`, `:707–728`). The runner caches its prepared command list onto `query.Context` so repeated runs reuse the optimized SQL when the statement is not parameter-dependent (`:274–283`).
- **Raw-SQL command surface (`CommandInfo` + `DataContextExtensions`).** `CommandInfo` is the entry point for raw SQL: ctors accept `(IDataContext, sql)`, `(…, sql, params DataParameter[])`, `(…, sql, DataParameter)`, or `(…, sql, object parameters)` (the last reflects mapped columns into `DataParameter[]` via a compiled lambda, `Source/LinqToDB/Data/CommandInfo.cs:72–123`, `:1657–1764`). The class enforces that `_dataContext` is `DataConnection` or `DataContext` (`:74–79`), routing to `GetDataConnection()` for execution. Public surface: `Query<T>(Func<DbDataReader,T>)`, `Query<T>()`, `QueryProc<T>(...)`, `QueryAsync*`, `QueryToListAsync`/`QueryToArrayAsync`/`QueryForEachAsync`/`QueryToAsyncEnumerable`, `QueryMultiple<T>`/`QueryProcMultiple<T>` (multi-resultset materialization keyed by `[ResultSetIndex]` attributes or member declaration order, `:779–877`), `Execute`/`Execute<T>`/`ExecuteAsync*`, `ExecuteReader`/`ExecuteReaderAsync`. Object readers are cached in a static `MemoryCache<QueryKey,Delegate>` keyed by `(targetType, readerType, configId, sql, extraKey, isScalar, scalarSourceType)` and cleared via `CommandInfo.ClearObjectReaderCache()` (`:1782–1793`). `DataContextExtensions` is a flat extension wrapper that creates a `CommandInfo` and forwards — every overload variant (parameter-form, async, template-typed, proc-typed) gets its own static method (`Source/LinqToDB/Data/DataContextExtensions.cs:31–2371`).
- **Bulk copy.** Public surface lives entirely on `DataContextExtensions` (`BulkCopy<T>` and `BulkCopyAsync<T>` with overloads accepting `BulkCopyOptions` / `int maxBatchSize` / nothing on either `IDataContext` or `ITable<T>`, `Source/LinqToDB/Data/DataContextExtensions.cs:2387–2823`). Each entry point delegates to `IDataProvider.BulkCopy(...)` after merging local options into `DataOptions` via `DataOptions.WithOptions(options)` and starts an `ActivityID.BulkCopy` activity (`:2392`, `:2415`, `:2435`). The actual bulk-copy strategies (`RowByRowBulkCopy`, `MultipleRowsBulkCopy`, provider-specific paths) live under `Source/LinqToDB/Internal/DataProvider/**` and are out of DATA scope. The DATA-side option bag is `BulkCopyOptions` — a sealed `record` over 21 parameters (`MaxBatchSize`, `BulkCopyTimeout`, `BulkCopyType`, `KeepIdentity`, `TableLock`, `KeepNulls`, `FireTriggers`, `UseInternalTransaction`, `ServerName`/`DatabaseName`/`SchemaName`/`TableName`, `TableOptions`, `NotifyAfter`+`RowsCopiedCallback`, `UseParameters`, `MaxParametersForBatch`, `MaxDegreeOfParallelism`, `WithoutSession`, `ConflictAction`) plus a v6→v7 binary-compat ctor and an explicit copy ctor (`Source/LinqToDB/Data/BulkCopyOptions.cs:134–238`). `BulkCopyType` is the four-way enum `Default`/`RowByRow`/`MultipleRows`/`ProviderSpecific` (`Source/LinqToDB/Data/BulkCopyType.cs:8–29`). `BulkCopyRowsCopied` is the callback DTO (`Abort`, `RowsCopied`, `StartTime`, `Source/LinqToDB/Data/BulkCopyRowsCopied.cs:5–21`). `ConflictAction` is the new `Default`/`Ignore` flag added for MySQL/Postgres/SQLite `INSERT … ON CONFLICT` lowering (`Source/LinqToDB/Data/ConflictAction.cs:6–23`).
- **Tracing.** `TraceInfo` carries every event (`DataConnection`, `Command`, `StartTime`, `ExecutionTime`, `RecordsAffected`, `Exception`, `MapperExpression`, `IsAsync`, lazy `SqlText` formatter, `Source/LinqToDB/Data/TraceInfo.cs:14–158`). Step is one of `BeforeExecute`/`AfterExecute`/`Error`/`MapperCreated`/`Completed` (`Source/LinqToDB/Data/TraceInfoStep.cs`). Operation is one of `ExecuteNonQuery`/`ExecuteReader`/`ExecuteScalar`/`BulkCopy`/`Open`/`BuildMapping`/`DisposeQuery`/`BeginTransaction`/`CommitTransaction`/`RollbackTransaction`/`DisposeTransaction` (`Source/LinqToDB/Data/TraceOperation.cs`). The trace pipeline is invoked from `DataConnection.OnTraceConnection` (default `OnTraceInternal` formats via `Pools.StringBuilder`, `Source/LinqToDB/Data/DataConnection.cs:593–695`) and from `TraceAction`/`TraceActionAsync` wrappers around transaction calls (`:1663–1718`, `Source/LinqToDB/Data/DataConnection.Async.cs:399–455`). Static `DataConnection.TraceSwitch` defaults to `Warning` in DEBUG and `Off` in Release (`Source/LinqToDB/Data/DataConnection.cs:697–704`). `QueryTraceOptions` is the option record that overrides `OnTraceConnection`/`WriteTraceLineConnection`/`TraceSwitchConnection` per `DataOptions` (`Source/LinqToDB/Data/QueryTraceOptions.cs:21–96`).
- **Transactions.** `DataConnectionTransaction` is the disposable controller returned from `BeginTransaction[Async]` — wraps `DataConnection` and tracks a `_disposeTransaction` flag so `Commit`/`Rollback` short-circuit dispose (`Source/LinqToDB/Data/DataConnectionTransaction.cs:10–77`). The transaction wrapper itself (`IAsyncDbTransaction`) is owned by `_connection` via `Internal.Async.AsyncFactory.CreateAndSetDataContext`. Begin/Commit/Rollback/Dispose all go through `TraceAction[Async]` so each transaction op produces a paired `BeforeExecute`/`AfterExecute`/`Error` trace (`DataConnection.cs:1495–1659`).
- **Retry policy.** `IRetryPolicy` exposes `Execute<T>(Func<T>)`, `Execute(Action)`, `ExecuteAsync<T>`, `ExecuteAsync` (`Source/LinqToDB/Data/RetryPolicy/IRetryPolicy.cs:7–46`). `RetryPolicyBase` is an EF Core-derived implementation (per the `BASEDON:` source comment at `Source/LinqToDB/Data/RetryPolicy/RetryPolicyBase.cs:1`) with exponential backoff, jitter via `RandomFactor`, and an `AsyncLocal<bool> _suspended` flag to prevent recursive retries (`:77–87`). Concrete provider policies (`SqlServerRetryPolicy`, `ClickHouseRetryPolicy`, `YdbRetryPolicy`) live under their `PROV-*` areas; `DefaultRetryPolicyFactory.GetRetryPolicy` is the single dispatch site that picks one based on `dataContext.DataProvider` type (`Source/LinqToDB/Data/RetryPolicy/DefaultRetryPolicyFactory.cs:11–46`). `TransientRetryPolicy` is the framework-provided cross-provider option that delegates to `DbException.IsTransient` and is gated `#if ADO_IS_TRANSIENT` (.NET 6+, `Source/LinqToDB/Data/RetryPolicy/TransientRetryPolicy.cs:3,15–58`); the same TFM gate guards `DbExceptionTransientExceptionDetector` (`Source/LinqToDB/Data/RetryPolicy/DbExceptionTransientExceptionDetector.cs:1–20`). `RetryingDbConnection` and `RetryingDbCommand` are the wrappers `DataConnection.GetOrCreateConnection()` installs around the underlying `IAsyncDbConnection` whenever `RetryPolicy != null` (`Source/LinqToDB/Data/DataConnection.cs:817–821`); they proxy every read/write op through `_policy.Execute[Async]` (`Source/LinqToDB/Data/RetryPolicy/RetryingDbCommand.cs:84–151`, `Source/LinqToDB/Data/RetryPolicy/RetryingDbConnection.cs:48–54`). `RetryLimitExceededException : LinqToDBException` is the surfaced failure type (`Source/LinqToDB/Data/RetryPolicy/RetryLimitExceededException.cs:6–14`). `RetryPolicyOptions` is the option record consumed by `ConfigurationApplier.Apply(DataConnection, RetryPolicyOptions)` (`Source/LinqToDB/Data/RetryPolicy/RetryPolicyOptions.cs:30–96`, applier at `Source/LinqToDB/Data/DataConnection.Configuration.cs:749–765`). Defaults: `MaxRetryCount = 5`, `MaxDelay = 30s`, `RandomFactor = 1.1`, `ExponentialBase = 2`, `Coefficient = 1s` (`RetryPolicyOptions.cs:96–102`).
- **Reader materialization.** `DataReaderWrapper` is the disposable that owns the `DbDataReader` + `DbCommand` returned from `ExecuteReader` and chains `BeforeReaderDispose` interceptor + `OnBeforeCommandDispose` + `_dataConnection.DataProvider.DisposeCommand(Command)` + optional `Close` when `IDataContext.CloseAfterUse` is set (`Source/LinqToDB/Data/DataReaderWrapper.cs:14–98`). `DataReaderAsync` is the higher-level `IDisposable`/`IAsyncDisposable` returned from `DataContextExtensions.ExecuteReader[Async]` — wraps `DataReaderWrapper`, tracks `ReadNumber` for multi-resultset navigation, exposes `Query<T>`/`QueryToListAsync`/`Execute<T>` etc. and emits a `TraceInfoStep.Completed` on dispose (`Source/LinqToDB/Data/DataReaderAsync.cs:14–303`).
- **`DataParameter`.** Public DTO for ADO parameters (`[ScalarType]`-marked, `Source/LinqToDB/Data/DataParameter.cs:14–270`). Stores `Name`, `Value`, `DbDataType` (lazy), `Direction`, `IsArray`, `Output` (filled by `CommandInfo.RebindParameters` for Output/InputOutput/ReturnValue parameters, `CommandInfo.cs:1633–1649`). Has a flat catalog of static factories per `DataType` (`Char`, `VarChar`, `Int32`, `DateTime2`, `DateTimeOffset`, `Json`, `BinaryJson`, …) plus a `Create`-suffixed overload set keyed on .NET source type — these are the canonical entry points users construct parameters with.
- **Configuration applier surface.** `ConnectionOptions` is the central record holding everything `DataConnection` needs to bind to a real `DbConnection`: `ConfigurationString`, `ConnectionString`, `IDataProvider`, `ProviderName`, `MappingSchema`, an existing `DbConnection` or `DbTransaction`, `DisposeConnection` flag, `ConnectionFactory` lambda, `DataProviderFactory` lambda, `ConnectionInterceptor`, `OnEntityDescriptorCreated` callback (`Source/LinqToDB/Data/ConnectionOptions.cs:53–127`). It implements `IApplicable<DataConnection>`/`IApplicable<DataContext>`/`IApplicable<RemoteDataContextBase>` so the same record drives all three context kinds (`:131–144`). `ConfigurationApplier.Apply(DataConnection, ConnectionOptions)` is a 7-tuple switch over `(ConfigurationString, ConnectionString, dataProvider, ProviderName, DbConnection, DbTransaction, ConnectionFactory)` that resolves the right binding path and caches the result back as `Saved*` fields on the options instance (`DataConnection.Configuration.cs:504–690`). `ConfigurationApplier.Reapply` enforces "not changeable dynamically" rules — only `MappingSchema` and `ConnectionInterceptor` can swap on a live context; everything else throws `LinqToDBException` (`:692–747`).

## Key types

- `DataConnection` — the persistent ADO.NET context (`Source/LinqToDB/Data/DataConnection.cs:32`). `partial`, six files. Implements `IDataContext` + `IInfrastructure<IServiceProvider>`. `Options` is the immutable `DataOptions` snapshot; `DataProvider` is the resolved `IDataProvider`; `_connection` is an `IAsyncDbConnection` (potentially wrapped in `RetryingDbConnection`); `_command` is a single shared `DbCommand` reused across executions and disposed via `DataProvider.DisposeCommand`.
- `CommandInfo` — `[PublicAPI]` raw-SQL command wrapper (`Source/LinqToDB/Data/CommandInfo.cs:43`). Mutable public fields `CommandText`, `Parameters`, `CommandType`, `CommandBehavior`. The static `MemoryCache<QueryKey,Delegate> _objectReaders` is the single cache point for all reader → object materialization in DATA (`:1782`).
- `DataConnectionTransaction` — `IDisposable`/`IAsyncDisposable` controller for transactions started via `DataConnection.BeginTransaction[Async]` (`Source/LinqToDB/Data/DataConnectionTransaction.cs:10`).
- `DataContextExtensions` — `[PublicAPI] public static class` hosting **all** raw-SQL and bulk-copy public methods. Receivers: `IDataContext` (most) and `ITable<T>` (bulk-copy). The class is roughly 2840 lines because every (sync/async × parameter-form × proc-flag × template-typed) combination gets its own overload — this is intentional shape to keep call-site type inference clean (`Source/LinqToDB/Data/DataContextExtensions.cs:21`).
- `DataParameter` — `[ScalarType]`-marked public DTO for ADO parameters (`Source/LinqToDB/Data/DataParameter.cs:15`).
- `DataReaderWrapper` — internal-construction disposable wrapping `DbDataReader` + `DbCommand` + before-dispose hook + optional `CloseAfterUse` close (`Source/LinqToDB/Data/DataReaderWrapper.cs:14`).
- `DataReaderAsync` — public `IDisposable`/`IAsyncDisposable` returned from `ExecuteReader[Async]`; the only public type that lets a caller iterate a multi-resultset reader by hand (`Source/LinqToDB/Data/DataReaderAsync.cs:14`).
- `BulkCopyOptions`, `BulkCopyType`, `BulkCopyRowsCopied`, `ConflictAction` — the public bulk-copy option surface (records + enums + a callback DTO).
- `ConnectionOptions`, `RetryPolicyOptions`, `QueryTraceOptions` — sealed `record` option types implementing `IOptionSet` + `IApplicable<DataConnection>` + `IReapplicable<…>`. They are composed inside `DataOptions` via the `OptionsContainer<T>` pattern documented in [CORE](../CORE/INDEX.md#recurring-patterns).
- `IRetryPolicy`, `RetryPolicyBase`, `TransientRetryPolicy`, `RetryingDbConnection`, `RetryingDbCommand`, `RetryLimitExceededException`, `DefaultRetryPolicyFactory`, `DbExceptionTransientExceptionDetector` — the entire retry-policy framework. `DefaultRetryPolicyFactory` is `internal static`; everything else is public.
- `LegacyMergeExtensions` — `[Obsolete]`-marked extension class for the pre-fluent merge API. Whole class is deprecated and slated for removal (`Source/LinqToDB/Data/LegacyMergeExtensions.cs:16`); the file is preserved as the documented migration-translation reference for users on the old API.
- `TraceInfo`, `TraceInfoStep`, `TraceOperation` — public tracing primitives.

## Recurring patterns

- **Sync/async sibling methods.** Every execution method (`ExecuteNonQuery`, `ExecuteScalar`, `ExecuteReader`, `BeginTransaction`, `CommitTransaction`, …) has a paired sync method in `DataConnection.cs` and async method in `DataConnection.Async.cs`. The async version is hand-written and structurally identical, with `await … .ConfigureAwait(false)` everywhere and `ActivityService.StartAndConfigureAwait` for activity spans. The same pattern is repeated inside `DataConnection.QueryRunner` (`ExecuteNonQueryImpl`/`ExecuteNonQueryImplAsync`, `ExecuteScalarImpl`/`ExecuteScalarImplAsync`, `ExecuteReader`/`ExecuteReaderAsync`); the source carries explicit `// In case of change the logic of this method, DO NOT FORGET to change the sibling method.` comments at every pair (`Source/LinqToDB/Data/DataConnection.QueryRunner.cs:362, 407, 452, 467, 481, 510, 557, 573, 626, 640`). New execution paths must add the sibling and the comment.
- **Trace wrapper triplet.** Every traceable op runs `BeforeExecute` → action → `AfterExecute`/`Error` via `TraceAction`/`TraceActionAsync` (`DataConnection.cs:1663–1718`, `DataConnection.Async.cs:399–455`). The wrapper formats a SQL string lazily via `commandText?.Invoke(context)` only when `TraceSwitchConnection.TraceInfo` is set, so `TraceLevel.Off` paths pay nothing. New transaction-style operations should reuse this wrapper rather than inline-tracing.
- **Interceptor-then-fallback.** Inside command-execute methods the pattern is "if `IInterceptable<ICommandInterceptor>.Interceptor` is set, ask it first, return its `Option<T>` if `HasValue`, else fall through to the raw provider call" (`DataConnection.cs:1082–1095`, `:1264–1276`, `:1356–1380`, mirrored in `DataConnection.Async.cs`). Any new public execute method must emit the same pattern for command-, exception-, and reader-related interceptors.
- **`CheckAndThrowOnDisposed` discipline.** Every public/internal method that mutates or reads connection-bound state starts with `CheckAndThrowOnDisposed()` (`DataConnection.cs:1078, 1106, 1167, 1349, 1397, 1497, 1538, 1574, 1607, 1640, 1740, 1753, 1767, 1817, 1825, …`). The flag is set to `true` only inside `Dispose()` / `DisposeAsync()`. The wiki-link error message (`https://github.com/linq2db/linq2db/wiki/Managing-data-connection`) is the single user-visible diagnostic for double-dispose.
- **Obsolete-with-`v7`-marker convention** (matches CORE). Every legacy ctor and several legacy properties on `DataConnection` carry `[Obsolete("This API scheduled for removal in v7. Instead use: …"), EditorBrowsable(EditorBrowsableState.Never)]` plus a `// TODO: Remove in v7` comment one line above. Examples: `DataConnection.cs:46–48`, `:74–76`, `:92–94`, `:783–785`, `:1043–1045`, `:1062–1064`, `:1786–1788`. The 16 ctor overloads at the top of `DataConnection.cs:43–482` are all obsoleted in favor of `DataConnection(DataOptions)` (`:489`).
- **`record` option types with `ConfigurationID` cache.** Every option type (`ConnectionOptions`, `BulkCopyOptions`, `QueryTraceOptions`, `RetryPolicyOptions`) is a sealed `record`, has a private copy ctor that re-copies every parameter (because record-with-expression-syntax doesn't carry custom-cached fields), implements `IConfigurationID.ConfigurationID` via a one-shot `IdentifierBuilder` that hashes every parameter, and overrides `Equals`/`GetHashCode` to compare on `ConfigurationID`. The "If you add another parameter here, don't forget to update copy ctor and IConfigurationID.ConfigurationID" comment is repeated on every record (`BulkCopyOptions.cs:158`, `ConnectionOptions.cs:69`, `QueryTraceOptions.cs:28`, `RetryPolicyOptions.cs:40`). New options must follow the same shape.
- **`ConfigurationApplier` Apply/Reapply pair.** Same pattern as [CORE](../CORE/INDEX.md#recurring-patterns)'s `DataContext.ConfigurationApplier`. `Apply(DataConnection, TOption)` mutates the live connection; `Reapply(DataConnection, TOption, prev)` returns an `Action?` undo delegate consumed by `UseOptions(...)` (`DataConnection.cs:1824–1851`). Lockstep changes to `DataConnection.Configuration.cs` and the option's `IReapplicable<DataConnection>.Apply` are required when adding option fields.

- **Sixteen obsoleted `DataConnection` ctors slated for v7 removal.** Lines 39–482 of `DataConnection.cs` are almost entirely obsoleted ctor overloads. They forward to `DataConnection(DataOptions)` after constructing an appropriate `DefaultDataOptions.UseX(...)` chain. When v7 lands these go away — any new feature that touches ctors should be added to `DataConnection(DataOptions)` only, never to the legacy overloads.
- **Several public properties retain obsoleted setters that are scheduled to become private in v7.** `RetryPolicy`, `OnTraceConnection`, `TraceSwitchConnection`, `WriteTraceLineConnection`, `IsMarsEnabled`, `Connection`, `CreateCommand`, `DisposeCommand`, `DisposeCommandAsync`, `EnsureConnectionAsync`, `ClearObjectReaderCache`, `ThrowOnDisposed` (`DataConnection.cs:541, 586, 741, 765, 558, 784, 1044, 1064, 1462, 1787`, `DataConnection.Async.cs:22, 148`). Internal callers wrap the use in `#pragma warning disable CS0618 / restore CS0618` — that pattern is everywhere in the area; new internal call sites should suppress with a comment, not migrate to the planned-private member by reflection.
- **`LegacyMergeExtensions` is class-level `[Obsolete]`.** The whole 745-line file is deprecated; the canonical message points users to the modern fluent merge API and to the file itself as a translation reference (`LegacyMergeExtensions.cs:16`). Do not extend this file.
- **`CommandInfo.ClearObjectReaderCache` is a globally-shared cache point.** The static `_objectReaders`, `_dataReaderConverter`, `_parameterReaders` `MemoryCache` instances are process-wide and not partitioned by `DataConnection` instance (`CommandInfo.cs:1782–1793`). Tests or hot-reload scenarios that swap `MappingSchema` / `IDataProvider` configurations need to call `ClearObjectReaderCache` to evict stale compiled mappers — there is no per-context invalidation. Note: `DataConnection.ClearObjectReaderCache` (`DataConnection.cs:1462–1466`) is itself `[Obsolete]` and forwards here.
- **Single shared `_command` field reused across executions.** `DataConnection._command` is a `DbCommand?` that survives between executions and is recycled by `InitCommand` / `DisposeCommand` (`DataConnection.cs:934–960`). After `ExecuteReader`, the command is detached (`_command = null`, `:1383`, `:702` async) and ownership transfers to `DataReaderWrapper`. New execution-path code must respect this lifecycle — leaving `_command` non-null when handing off a reader leaks the command on the next `InitCommand` rebuild.
- **Multi-statement DROP-then-X loop swallows DROP exceptions silently.** `DataConnection.QueryRunner.ExecuteNonQueryImpl[Async]` runs each statement in a multi-command batch in order, but if an intermediate statement begins with `"DROP"` and fails, the exception is caught and discarded so the rest of the batch can run (`Source/LinqToDB/Data/DataConnection.QueryRunner.cs:382–392`, `:423–432`, `:713–722`). This is intentional (drop-if-exists semantics) but obscures genuine permission/state errors. Any new pre-execute statement that should be best-effort needs to use the same `StartsWith("DROP", StringComparison.Ordinal)` marker; arbitrary best-effort SQL is not supported.
- **`TransientRetryPolicy` is gated behind `ADO_IS_TRANSIENT` (.NET 6+ only).** The `#if ADO_IS_TRANSIENT` guard at the top of both `TransientRetryPolicy.cs` and `DbExceptionTransientExceptionDetector.cs` means net462/netstandard2.0 callers cannot use the cross-provider transient-retry policy and must pick a provider-specific retry policy or implement `IRetryPolicy` themselves. The file comment notes that some providers (MySqlConnector) implement `DbException.IsTransient` on older runtimes but are not handled (`TransientRetryPolicy.cs:1–2`).
