# DI-0603 -- Commented-out dead code in InformixMemberTranslator (incomplete week-number and millisecond DateAdd)

**Severity**: low  **Category**: commented-out-code  **Status**: open

## Location
- `Source/LinqToDB/Internal/DataProvider/Informix/Translation/InformixMemberTranslator.cs:179` -- commented-out `result = factory.Cast(Cast(result, char(10)), int)` block in `Week` DatePart branch
- `Source/LinqToDB/Internal/DataProvider/Informix/Translation/InformixMemberTranslator.cs:275` -- commented-out `fragmentStr = "Second to Fraction (3)"` block in `Millisecond` DateAdd branch, labelled "Non working code"

## Pattern matched
Commented-out code blocks (`/* ... */`) containing non-trivial logic that is intentionally disabled. Both blocks are multi-line and contain production-level expressions that were either abandoned mid-refactor or found non-functional and left in place rather than being removed or replaced with a proper fallback.

## Why it matters
The `Week` branch leaves a finalising cast step disabled -- the intermediate interval arithmetic result is returned without the `::char(10)::int` coercion that converts it to a plain integer, which may produce an unexpected Informix INTERVAL value rather than an INT to callers. The `Millisecond` branch explicitly labels itself "Non working code" and returns `null`, meaning `DateAdd` for millisecond granularity silently falls back to no translation for Informix; any user code that relies on millisecond precision in date arithmetic will produce a runtime "cannot convert" error rather than a clear unsupported-operation message. Retaining the dead code increases maintenance cost and misleads future contributors into thinking the feature is nearly complete.

## Suggested fix
- `Week` branch: verify whether the cast to `char(10)` then `int` is actually required by Informix to return an integer from interval division; if yes, uncomment and fix; if no, remove the dead block.
- `Millisecond` branch: remove the commented fragment; throw `LinqToDBException("Millisecond DateAdd is not supported by Informix")` or leave only the `return null` with an explanatory single-line comment. Track the gap in a GitHub issue if millisecond support is planned.
