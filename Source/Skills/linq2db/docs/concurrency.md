# Optimistic Concurrency

> ⚠️ **Stop. This document is incomplete by itself.**
> Before implementing anything, read [`SKILL.md`](../SKILL.md).
> It contains global rules, required namespaces, architecture constraints, and documentation navigation.
> Do not continue without reading it.

Use this guide when updating or deleting one entity instance should include its current optimistic
lock column value in the SQL filter.

The optimistic concurrency API is in the `LinqToDB.Concurrency` namespace:

```csharp
using LinqToDB.Concurrency;
using LinqToDB.Mapping;
```

## API Surface

| Scenario | API | Receiver | Result |
|---|---|---|---|
| Update entity through context table | `UpdateOptimistic(entity)` | `IDataContext` | number of updated records |
| Update entity through filtered source | `UpdateOptimistic(entity)` | `IQueryable<T>` | number of updated records |
| Delete entity through context table | `DeleteOptimistic(entity)` | `IDataContext` | number of deleted records |
| Delete entity through filtered source | `DeleteOptimistic(entity)` | `IQueryable<T>` | number of deleted records |
| Build optimistic key filter | `WhereKeyOptimistic(entity)` | `IQueryable<T>` | filtered `IQueryable<T>` |

Async forms are available for update and delete:

- `UpdateOptimisticAsync(entity, cancellationToken)`
- `DeleteOptimisticAsync(entity, cancellationToken)`

The entity type must be a class. The entity mapping must have a primary key; the implementation
throws a `LinqToDBException` when no primary key is defined.

## Mark The Lock Column

Mark the version/lock column with `OptimisticLockPropertyAttribute`.

```csharp
using LinqToDB.Mapping;

[Table("Products")]
public sealed class Product
{
    [PrimaryKey]
    public int Id { get; set; }

    [Column(Length = 200)]
    public string Name { get; set; } = null!;

    [OptimisticLockProperty(VersionBehavior.AutoIncrement)]
    public int Version { get; set; }
}
```

XML-doc states that `OptimisticLockPropertyAttribute` implements built-in optimistic lock value
generation strategies and is used with `LinqToDB.Concurrency.ConcurrencyExtensions`.

Supported built-in `VersionBehavior` values:

| Value | Meaning |
|---|---|
| `Auto` | Column value is generated by the database automatically on update, for example SQL Server `rowversion`/`timestamp` or a trigger. LinqToDB skips explicit version column assignment. |
| `AutoIncrement` | Column value should be incremented by `1`. |
| `Guid` | Uses `Guid.NewGuid()`; supported column types are `Guid`, `string`, and `byte[]`. |

For custom lock-column generation, derive from `OptimisticLockPropertyBaseAttribute` and implement
`GetNextValue(...)`. XML-doc says returning `null` means the new value is generated by the
database.

## Update With Optimistic Lock

Use `UpdateOptimistic` for an entity instance that contains the original lock value.

```csharp
using LinqToDB.Concurrency;

product.Name = newName;

var affected = db.UpdateOptimistic(product);
```

The generated operation filters by primary key and by columns annotated with
`OptimisticLockPropertyBaseAttribute`. It updates mapped non-primary-key, non-identity,
non-skipped columns. For annotated lock columns, LinqToDB uses the configured lock-column
generation strategy for the new value.

The return value is the number of updated records. If another transaction changed the lock column
before this update, the optimistic filter should match no row and the affected count should be `0`.

Async:

```csharp
using LinqToDB.Concurrency;

var affected = await db.UpdateOptimisticAsync(product, cancellationToken);
```

## Update Through A Filtered Source

Use the `IQueryable<T>` receiver when the update must also be constrained by an existing table
source/filter.

```csharp
using LinqToDB.Concurrency;

var affected = db.GetTable<Product>()
    .Where(p => p.TenantId == tenantId)
    .UpdateOptimistic(product);
```

The `source` must be a LinqToDB query. The implementation retrieves the LinqToDB data context from
the source and throws an `ArgumentException` when the source is not a LinqToDB query.

## Delete With Optimistic Lock

Use `DeleteOptimistic` when deleting one entity instance should also check the lock column.

```csharp
using LinqToDB.Concurrency;

var affected = db.DeleteOptimistic(product);
```

Async:

```csharp
using LinqToDB.Concurrency;

var affected = await db.DeleteOptimisticAsync(product, cancellationToken);
```

The same `IQueryable<T>` receiver shape is available:

```csharp
using LinqToDB.Concurrency;

var affected = db.GetTable<Product>()
    .Where(p => p.TenantId == tenantId)
    .DeleteOptimistic(product);
```

## Build The Optimistic Filter

Use `WhereKeyOptimistic(entity)` when you need the primary-key + optimistic-lock filter as a query
instead of immediately updating or deleting:

```csharp
using LinqToDB.Async;
using LinqToDB.Concurrency;

var current = await db.GetTable<Product>()
    .WhereKeyOptimistic(product)
    .SingleOrDefaultAsync();
```

XML-doc states that if no optimistic lock column is annotated, `WhereKeyOptimistic` applies only
the primary key filter.

## No Lock Attribute Means Regular Operation

XML-doc states that:

- `UpdateOptimistic` performs a regular update when the entity has no column annotated with
  `OptimisticLockPropertyBaseAttribute`;
- `DeleteOptimistic` performs a regular delete when the entity has no column annotated with
  `OptimisticLockPropertyBaseAttribute`;
- `WhereKeyOptimistic` applies only the primary key filter when there is no annotated lock column.

Do not claim optimistic concurrency protection exists unless the entity mapping contains an
optimistic lock attribute.

## Relationship To Set-Based Updates

`UpdateOptimistic(entity)` is for one mapped entity instance with primary key and lock-column
values. For set-based updates that update all rows matching a predicate, use the normal update APIs
from [`crud/crud-update.md`](crud/crud-update.md) and explicitly include whatever concurrency
predicate your application requires.

## Common Mistakes

### Missing namespace

Wrong:

```csharp
var affected = db.UpdateOptimistic(product);
```

Correct:

```csharp
using LinqToDB.Concurrency;

var affected = db.UpdateOptimistic(product);
```

### No primary key mapping

Wrong:

```csharp
public sealed class Product
{
    public int Id { get; set; }

    [OptimisticLockProperty(VersionBehavior.AutoIncrement)]
    public int Version { get; set; }
}
```

Correct:

```csharp
public sealed class Product
{
    [PrimaryKey]
    public int Id { get; set; }

    [OptimisticLockProperty(VersionBehavior.AutoIncrement)]
    public int Version { get; set; }
}
```

### Assuming `UpdateOptimistic` throws on a stale version

Wrong:

```csharp
await db.UpdateOptimisticAsync(product);
// assume success
```

Correct:

```csharp
var affected = await db.UpdateOptimisticAsync(product);

if (affected == 0)
{
    // stale version, missing row, or an additional source filter did not match
}
```

The package API returns the affected row count. Do not document exception behavior for stale
versions unless a version-matched XML-doc/API entry explicitly states it.
